\section{Problem Definition}
\label{sec:problem_definition}

To construct a high-performance local dashboard for food stall operators, the problem must be formally defined in terms of its operational bounds, input schemas, output requirements, and success metrics. The Vendor Engine functions as a local, concurrent mediator that decouples the network-dependent order transmission from real-time kitchen operations.

\subsection{Inputs to the System}
The application ingests several distinct inputs to manage stall configurations and order flows:
\begin{itemize}
    \item \textbf{Stalls Configuration File (\code{stalls.json})}: A static JSON configuration file loaded at application startup defining registered food stalls. It maps unique stall IDs to human-readable names.
    \item \textbf{Simulated WebSocket Order Stream}: A high-throughput stream of JSON order payloads generated by the simulation layer. Each order includes:
    \begin{itemize}
        \item \code{orderId}: A unique string representation of a UUID.
        \item \code{stallId}: The ID of the target stall.
        \item \code{priority}: The priority tier of the order (\code{VIP} or \code{STANDARD}).
        \item \code{createdAt}: A Unix millisecond timestamp representing when the order was placed.
        \item \code{items}: A JSON array of line items, where each item contains \code{itemName}, \code{quantity}, and \code{unitPrice}.
    \end{itemize}
    \item \textbf{Heartbeat Connectivity Signal (\code{network.flag})}: A control file polled periodically by the system's background network monitor. The file contains a single string, either \code{"up"} or \code{"down"}, simulating WebSocket network connectivity.
    \item \textbf{User Authentication Credentials}: User-selected roles (\code{KITCHEN\_}\allowbreak\code{WORKER} or \code{MERCHANT\_}\allowbreak\code{ADMIN}) and assigned stall IDs supplied via the graphical login screen.
\end{itemize}

\subsection{Outputs of the System}
The application generates several visible and persistent outputs:
\begin{itemize}
    \item \textbf{Live Prioritized Kanban Boards}: Java Swing-based graphical displays showing sorted active orders in three status-based columns (Pending, Accepted, Ready to Serve).
    \item \textbf{Offline Binary State Backup (\code{offline\_}\allowbreak\code{stalls.ser})}: A serialized binary snapshot of the entire application state (including order queues and configurations) written to disk automatically when network loss is detected.
    \item \textbf{Reconnection Synchronization Payload (\code{sync\_}\allowbreak\code{payload.json})}: A structured JSON file containing all orders processed and served locally during an offline period, written upon network restoration to sync with the backend.
    \item \textbf{Real-Time Operational Analytics}: Dynamic visual metrics displayed to administrators, including total revenue in Indian Rupees (₹), total active queue depths, and live revenue trend charts.
\end{itemize}
